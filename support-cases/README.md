# How the Repository is Organized

Support incidents are organized **by sprint**.  
Each sprint has its own folder containing the support cases handled during that period.

Example structure:

```
support-cases/
│
├── sprint-99-support-cases/
│   ├── 2026-03-02-leads-sync-issue.md
│   ├── 2026-03-04-airtable-key-project-issue.md
│   ├── 2026-03-04-leads-slack-automation-redesign.md
│   ├── 2026-03-04-new-project-okrs-error.md

│   ├── 2026-03-05-profile-page-blank-issue.md

```

# Sprint Folder Naming Convention

Sprint folders follow this format:

```
sprint-[sprint-number]-support-cases
```

Example:

```
sprint-99-support-cases
```

This structure keeps all incidents grouped according to the sprint in which they were resolved.

---

# Support Case File Naming Convention

Each incident follows this format:

```
YYYY-MM-DD-issue-description.md
```

Where:

| Component | Description |
|--------|--------|
| YYYY-MM-DD | Date when the issue was handled |
| issue-description | Short description of the incident |

Example:

```
2026-03-05-profile-page-blank-issue.md
```

This format ensures that cases are **chronologically ordered and easy to locate**.

---

# Contents of Each Support Case

The current support documentation format is based on the structure used in the project template and sprint records.

Each support case should include:

- **Case / Title** – Name of the support incident  
- **Date** – When the incident occurred  
- **Context** – Brief explanation of the problem or request  
- **Actions Taken** – Steps performed to review or resolve the case  
- **Result** – Outcome after the work was completed  
- **Time Spent** – Effort invested in the case  

In consolidated sprint documents, the case may also include a numeric label such as `Caso 1`, `Caso 2`, etc.

This keeps the documentation aligned with the actual format currently used by the team.

This documentation allows the team to:

- Track support work completed during each sprint

- Record effort invested in support activities
